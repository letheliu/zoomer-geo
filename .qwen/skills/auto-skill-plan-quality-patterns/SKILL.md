# 计划文档质量模式

brainstorming 技能产出设计决策与 spec，writing-plans 技能产出实现计划骨架。但对于**多模块、多任务、跨阶段集成**的复杂计划，单靠这两个技能产出的"任务列表"在执行阶段容易踩坑：

- 任务改了既有代码但没标注会破坏什么 → 编译/测试失败但不知道为什么
- 类型在 spec 里定义，跨任务漂移用错字段名 → 后期集成才暴露
- spec 章节多，覆盖度靠记忆 → 漏实现某些需求

本技能沉淀三组补强模式：**TDD 任务 5 步结构**、**任务原子性风险警告**、**自检表（规格覆盖 / 类型一致性 / 风险）**。

## 何时使用

- 计划涵盖 ≥10 个任务
- 计划会修改既有阶段的代码（跨阶段集成）
- spec 文档章节多、含多个数据模型/接口契约
- 实施者可能是另一个 agent 或几天后的自己

如果只是 3-5 步的小计划，writing-plans 默认输出就够了，不需要这个技能。

## 模式 1：TDD 任务 5 步结构

每个任务必须按以下 5 步写，禁止用"实现 + 测试"两段式：

```markdown
## 任务 N：[组件名称]

**文件：**
- 创建：`src/path/to/file.ts`
- 测试：`src/path/to/file.test.ts`

- [ ] **步骤 1：编写失败的测试**

[完整测试代码，含所有 import 和 describe/it 块]

- [ ] **步骤 2：运行测试验证失败**

运行：`npx vitest run src/path/to/file.test.ts`
预期：FAIL，报错 "[specific reason]"

- [ ] **步骤 3：编写最少实现**

[完整实现代码，含所有 import 和完整函数体]

- [ ] **步骤 4：运行测试验证通过**

运行：`npx vitest run src/path/to/file.test.ts`
预期：PASS（[N] 个测试全过）

- [ ] **步骤 5：Commit**

```bash
git add src/path/to/file.ts src/path/to/file.test.ts
git commit -m "feat(module): [imperative summary]"
```
```

**Why：** 步骤 1-2 强制测试先写、强制验证失败（避免"写完测试已经过"的伪 TDD）。步骤 3 的"最少实现"避免过度设计。步骤 4 的"预期 PASS"给出可验证的完成标志。步骤 5 给每次 commit 明确的边界。

**How to apply：** 任何含测试的任务都按这个结构。纯文档/配置任务（无测试）可以省略步骤 1-2，但保留步骤 3-5。

## 模式 2：任务原子性风险警告

当某个任务**修改既有代码**且**该修改会让项目在那个 commit 之后编译/测试失败**时，必须在任务的步骤 5 commit 之后追加一个 **Risk Block**：

```markdown
> **风险警告：** 此 commit 后，[既有文件 X] 的 [位置/函数 Y] 会编译失败/行为变化，因为 [具体原因]。任务 M（[简述]）必须紧接着完成。推荐执行时把任务 N 和 M 视为原子提交组。
```

**典型触发场景：**
- 改了某个 service 的构造函数签名（如 `createXxx(prisma)` → `createXxx({ prisma })`），但调用方未同步改
- 重命名/删除一个被其他文件 import 的导出
- 改了 Prisma 模型但既有代码还引用旧字段名
- 改了 Router 签名但 server.ts 装配未同步

**Why：** 多 agent 串行执行时，下游任务执行者看到的是失败的编译状态，可能误判是新增的代码错。明确警告能让他直接跳到目标修复任务。

**How to apply：** 写完每个任务步骤 5 后，自问"此 commit 后，项目是否能 build？现有测试是否会失败？" 如果答案是"不能"或"会"，追加 Risk Block。

## 模式 3：自检表（计划末尾）

计划末尾必须有 3 个自检表：

### 表 A：规格覆盖度

| 规格章节 | 覆盖任务 |
|----------|---------|
| spec §5.3 组件 A | 任务 3, 7 |
| spec §5.4 数据模型 X | 任务 1, 5 |
| spec §6 接口契约 | 任务 12, 14 |

**用法：** 把 spec 每个章节/需求列一行，标注覆盖它的任务编号。如果有 spec 章节没对应任务，**补任务**。

### 表 B：类型一致性

| 类型/接口 | 定义处 | 引用处 | 一致性 |
|----------|--------|--------|--------|
| `MyType` | 任务 2（types.ts） | 任务 5（service 输入）、任务 8（router） | ✓ |
| `onEvent` 钩子 | 任务 9（service） | 任务 12（server.ts 装配） | ✓ |

**用法：** 把跨任务复用的关键类型/接口列出来。`定义处` 标哪个任务文件定义了它；`引用处` 列出所有用到的任务。一致性打勾表示跨任务签名/字段名匹配。

### 表 C：风险提示

| 风险 | 触发任务 | 缓解 |
|------|---------|------|
| 任务 9 改动 X 会让既有 Y 编译失败 | 任务 9 commit 后 | 任务 11 必须立即完成 |
| 任务 1 的 prisma generate 失败会阻塞任务 5/7/10 | 任务 1 步骤 5 | 执行前先运行 prisma generate |
| 第三方 API 限制 | 任务 7 | 需要在生产配置 API key |

**用法：** 集中列出所有任务级 Risk Block 中的风险，并加上非任务级的风险（如"第三方依赖失败"）。

## 三个模式的串联

完整的工作流是：

1. **写 spec**（brainstorming 技能产出）
2. **写计划骨架**（writing-plans 技能产出：文件结构 + 任务列表）
3. **应用本技能强化**：
   - 把每个任务的"实现 + 测试"步骤重写为 5 步 TDD 结构
   - 检查每个任务，必要时追加 Risk Block
   - 在计划末尾追加 3 个自检表
4. **执行前自检**：对照表 A 补漏、表 B 修正、表 C 在脑里走一遍

## 反模式（避免）

- ❌ 任务步骤里只写"添加测试"或"编写实现"不附代码 — 执行者无从下手
- ❌ 用 `// ... 更多测试` 这类省略 — 测试代码必须完整
- ❌ 任务粒度过粗（"实现整个模块"一个任务）— 违反小步 TDD
- ❌ 任务粒度过细（每个 if 分支一个任务）— 失去执行焦点
- ❌ 改既有代码不标 Risk Block — 必踩坑
- ❌ 自检表"看上去都覆盖了"但实际跑过代码 — 表 A 写完必须真的翻 spec 章节
- ❌ 表 B 只列类型名不列具体定义/引用位置 — 失去核对价值

## 与其他技能的关系

- **brainstorming**：本技能不重复其"问问题对齐"流程，但要求 spec 文档结构清晰（章节可被表 A 引用）
- **writing-plans**：本技能是其输出的"补强"层，不替代
- **executing-plans / subagent-driven-development**：本技能的 5 步任务结构和自检表都是为执行阶段服务的
